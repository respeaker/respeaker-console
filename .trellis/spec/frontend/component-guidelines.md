# Component Guidelines

> How components are built in this project.

---

## Overview

All components are function components with hooks. No class components. No HoCs. shadcn/ui provides the primitive layer; domain panels compose them.

---

## Component Structure

```typescript
// 1. Imports (React, UI, hooks, lib)
import { useState } from "react";
import { Button } from "@/components/ui/button";
import { useXvf } from "@/hooks/use-xvf";

// 2. Props interface (if needed)
interface DevicePanelProps {
  xvf: UseXvfResult;
}

// 3. Component (named export for panels, default export for pages)
export function DevicePanel({ xvf }: DevicePanelProps) {
  // hooks first, then handlers, then render
  return (...);
}
```

---

## Props Conventions

### Domain panels receive the controller hook result

```typescript
interface PanelProps {
  xvf: UseXvfResult;
}
```

### UI primitives use shadcn/ui conventions

- Extend native element props via `React.ComponentProps<"div">`
- Accept `className` and merge with `cn()`
- Use `forwardRef` only when needed for focus management

---

## Styling Patterns

- Tailwind CSS v4 classes exclusively
- Use `cn()` from `@/lib/utils` to merge conditional classes
- CSS variables for theme tokens (light/dark via `ThemeProvider`)
- No inline styles, no CSS modules, no styled-components

---

## Composition Rules

1. **Pages** own providers and layout chrome (`WindowFrame`, `ThemeProvider`, `Toaster`)
2. **Panels** own domain logic display; they call methods on the hook result
3. **UI components** are stateless presentation; state lives in hooks or panels

## Lazy Loading

Heavy panels (e.g., ConfigPanel with hundreds of parameters) are lazy-loaded:

```typescript
const ConfigPanel = lazy(() =>
  import("@/components/xvf/config-panel").then((m) => ({ default: m.ConfigPanel })),
);
```

## Pattern: SVG Instrument Visualization

For data visualization components (compass, gauges, meters):

- Use a fixed `viewBox` (e.g., `0 0 200 200`) with computed geometry via constants
- Keep all drawing logic in a single render — no child components for SVG internals
- Use Tailwind classes on SVG elements via `className` for theme-aware colors (`stroke-muted-foreground`, `fill-green-500`)
- Add `cn("transition-colors", ...)` on elements that change with state (e.g., VAD active/idle)
- Expose a minimal props interface: data value + state flag + optional `className`
- Cardinal labels, tick marks, and concentric rings give depth without decoration

```typescript
// Constants at module top, not inside render
const CX = 100;
const CY = 100;
const OUTER_R = 88;

export function Gauge({ value, active, className }: GaugeProps) {
  const rad = ((value - 90) * Math.PI) / 180;
  const color = active ? "stroke-green-500" : "stroke-muted-foreground/60";
  return (
    <svg viewBox="0 0 200 200" className={cn("h-48 w-48", className)}>
      {/* geometry computed from constants + props */}
    </svg>
  );
}
```

## Pattern: Dangerous Operation Confirmation

For write operations that are destructive or irreversible (reboot, save config, flash firmware):

- Use shadcn/ui `AlertDialog` with controlled `open` state
- Intercept the action handler, set dialog state, then execute only on confirm
- Use `variant="destructive"` on `AlertDialogAction` for destructive ops
- Style: `max-w-sm`, `font-medium` title, `leading-relaxed` description, `font-normal` buttons
- All user-facing text via i18n keys under `xvf.confirm.<operation>.*`

```typescript
const [confirmDialog, setConfirmDialog] = useState<"reboot" | "saveConfig" | null>(null);

const handleDangerous = () => setConfirmDialog("reboot");
const handleConfirm = () => {
  setConfirmDialog(null);
  void executeAction();
};

<AlertDialog open={confirmDialog !== null} onOpenChange={(o) => { if (!o) setConfirmDialog(null); }}>
  <AlertDialogContent className="max-w-sm">
    <AlertDialogAction variant="destructive" onClick={handleConfirm}>
      {t("xvf.confirm.reboot.confirm")}
    </AlertDialogAction>
  </AlertDialogContent>
</AlertDialog>
```

---

## Forbidden Patterns

- No class components or HoCs
- No prop drilling beyond 2 levels — pass the hook result directly
- No inline styles — use Tailwind classes
- No direct DOM manipulation — use refs only for focus/measurement
- Don't edit `components/ui/` logic — they are generated by shadcn CLI
